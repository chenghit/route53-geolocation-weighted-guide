# Route 53 多层路由方案对比：Traffic Policy vs 手动 CNAME 链

## 适用场景

当你需要在 Route 53 中组合多种路由策略时（例如：针对特定国家的用户按权重分流到不同 CDN），Route 53 提供两种实现方式。本文对比它们的差异，帮助你选择适合的方案。

典型场景举例：
- 特定国家的用户 50/50 分流到 Akamai 和 CloudFront
- 按地区分流后，再按权重灰度切换新旧源站
- 地理位置路由 + 加权路由的任意组合

---

## 方案概览

| | **方案 A：Traffic Policy** | **方案 B：手动 CNAME 链** |
|---|---|---|
| 原理 | 一个 JSON 文档定义完整路由逻辑，Route 53 自动管理底层记录 | 手动创建两层 CNAME 记录：Geo 层 → 中间子域名 → Weighted 层 |
| 中间域名 | 不需要 | 需要（如 `weighted.example.com`） |
| 操作步骤 | 2 步（创建策略 + 创建 Instance） | 4+ 步（创建加权记录 → 创建地理位置记录，可选 Health Check） |
| 修改权重 | 创建新版本 → 更新 Instance（原子切换，可回滚） | UPSERT 直接覆盖（立即生效，无版本概念，无法回滚） |
| 可视化 | Route 53 控制台提供 Traffic Flow 可视化编辑器 | 无，只能查看原始记录列表 |

---

## 费用对比

| 费用项 | Traffic Policy | 手动 CNAME 链 |
|--------|---------------|--------------|
| **固定月费** | **$50/月/域名**（Instance 费用） | **$0** |
| DNS 查询（目标国家用户） | $0.70/百万次（Geo 档） | $0.70 + $0.40 = **$1.10/百万次**（两层查询） |
| DNS 查询（其他地区用户） | $0.70/百万次 | $0.70/百万次 |
| Health Check（可选） | $0.50–$0.75/月/个 | $0.50–$0.75/月/个 |

**盈亏平衡点**：Traffic Policy 每月多花 $50 固定费，但目标国家用户每次查询少花 $0.40/百万次。当目标国家月查询量超过约 **1.25 亿次** 时，Traffic Policy 反而更便宜。大多数场景达不到这个量级，因此 **Traffic Policy 的总成本通常更高**。

---

## 配额对比

| 配额项 | Traffic Policy | 手动 CNAME 链 |
|--------|---------------|--------------|
| **关键瓶颈** | **Instance 默认 5 个/账号**（可申请提升） | 无专属限制 |
| 策略/记录数量 | 50 个策略/账号，1,000 版本/策略 | 10,000 条记录/Hosted Zone |
| 同名同类型记录 | 由策略内部管理，不受 100 条限制 | Weighted/Geo 同名同类型最多 100 条 |
| 提升难度 | 需通过 Service Quotas 申请 | 记录数可申请，通常秒批 |

> **什么是 Instance？** Traffic Policy 本身只是一个路由规则模板。要让它对某个域名生效，需要创建一个 Instance 把模板"绑定"到域名上。**一个 Instance = 一个域名。** 默认 5 个意味着整个 AWS 账号最多只能对 5 个域名使用 Traffic Policy。超过 5 个需要提前申请提升配额。手动 CNAME 链没有这个限制。

## 运维体验对比

| 维度 | Traffic Policy | 手动 CNAME 链 |
|------|---------------|--------------|
| 修改权重 | 创建新版本 + 更新 Instance（2 步 API） | UPSERT 加权记录（1 步 API） |
| 回滚 | ✅ 更新 Instance 指向旧版本，秒级回滚 | ❌ 无版本概念，需手动 UPSERT 回旧值 |
| 版本历史 | ✅ 每个版本的 JSON 文档完整保留 | ❌ 无历史记录 |
| 自动化集成 | JSON 文档可纳入 Git 版本管理 | 需自行维护记录的期望状态 |
| 学习成本 | 需理解 Policy Document JSON 格式 | 只需理解基础 DNS 记录概念 |

---

## 推荐

**大多数生产场景推荐 Traffic Policy。**

理由：

1. **架构更干净**——不需要中间域名，Hosted Zone 里不会出现内部路由用的子域名记录
2. **版本管理 + 原子回滚**——改权重不怕改错，随时回滚到上一个版本
3. **操作步骤少**——2 步完成 vs 4+ 步，出错概率更低
4. **可审计**——策略定义是完整的 JSON 文档，可以放进 Git、做 code review、跨环境复用

$50/月的固定成本对于生产环境的流量管理来说通常可以接受。

---

## 什么时候用手动 CNAME 链

以下场景更适合手动方案：

1. **成本敏感 + 域名多**——如果有 20+ 个域名都需要相同的多层路由，Traffic Policy 要 $1,000+/月。虽然可以用"一个 Instance + 其他域名 CNAME 指向它"来省钱，但这引入了额外的 CNAME 跳转
2. **Instance 配额不够**——默认只有 5 个，申请提升需要时间。如果急着上线且配额不够，手动方案可以立即使用
3. **外部监控系统需要直接控制权重**——手动方案一步 UPSERT 即可改权重并立即生效。Traffic Policy 需要先创建新版本再更新 Instance（2 步），对于需要秒级自动切换的场景多了一步
4. **临时测试 / PoC**——不想为短期测试付 $50/月
